# Phase 12 Evidence Artifacts

## Evidence Packs

Phase 12 evidence artifacts are grouped into these packs:

| Pack | Purpose | Expected Artifacts | Status Semantics |
| --- | --- | --- | --- |
| detector-package-safe-boot | Detector, package identity, safe boot, and trusted wrapper capture | `detect-ultra205.log`, `safe-boot/flash-command-evidence.json`, `safe-boot/flash-monitor.log` | Supports safe-state evidence only; unsupported ASIC/mining tiers remain `hardware evidence pending`. |
| bm1366-chip-detect-init | BM1366 chip-detect and staged-init/no-mining diagnostic evidence | `chip-detect/flash-command-evidence.json`, `chip-detect/flash-monitor.log` | Supports only exact observed chip-detect or staged-init claims. |
| diagnostic-work-result | Typed diagnostic work-send and result-receive evidence | Plan-approved probe output | Unsupported until a repo-owned bounded probe generates artifacts. |
| controlled-mining-smoke | First controlled mining-loop smoke evidence | Plan-approved probe output and redacted logs | Unsupported until a bounded smoke path records share/no-share, telemetry, watchdog, and safe-stop details. |
| bounded-mining-soak | Bounded mining soak evidence | Plan-approved soak output and redacted logs | Unsupported until duration, stop conditions, telemetry, watchdog, share/no-share, and safe-stop details exist. |
| parity-promotion | Checklist and validation promotion proof | `just parity`, `just test`, Rust checks, checklist diff, validation report | Supports promotion only when rows match exact evidence. |

## Generated By Repo Commands

`flash-command-evidence.json` and `flash-monitor.log` are generated by the wrapper command:

```bash
just flash-monitor board=205 port=<path> evidence-dir=<path>
```

`detect-ultra205.log` is generated by capturing:

```bash
just detect-ultra205
```

Probe artifacts are generated only by a plan-approved Phase 12 probe. Do not invent diagnostic work, mining smoke, or soak artifacts by hand.

## Required Metadata

Each accepted hardware evidence pack must record:

- Board `205`.
- Selected port.
- Source commit.
- Reference commit.
- Package manifest or firmware identity when applicable.
- Exact command.
- Detector output and board-info output when hardware is used.
- Captured logs or probe output.
- Observed behavior.
- Redaction result.
- Conclusion.

## Pending Evidence Semantics

Unsupported tiers remain `hardware evidence pending`.

Pending evidence is a valid Phase 12 result when detector, recovery, safety, chip-detect, controlled pool, redaction, or bounded-probe prerequisites are missing. Pending status must not be rewritten as `verified` in `docs/parity/checklist.md`.

## Do Not Hand Edit Generated Artifacts

Do not hand edit generated JSON, captured serial logs, or probe output files.

If a generated artifact contains a secret or private value, keep the generated file unstaged, update `redaction-review.md` with the finding, and either rerun the command with a safer capture path or record the evidence pack as blocked by redaction review.
