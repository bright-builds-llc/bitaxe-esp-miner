# Phase 11 Safety Evidence Artifacts

## Component Evidence Packs

Evidence packs are component-scoped so one successful observation cannot overclaim a broader safety surface. Each pack may contain wrapper-generated artifacts, executor-written reviews, and a conclusion that applies only to the named component.

| Pack | Expected Artifacts | Conclusion Rules |
| --- | --- | --- |
| `safe-baseline` | `flash-command-evidence.json`, `flash-monitor.log`, `redaction-review.md`, optional detector transcript copied into the human ledger | May conclude passed only for board `205` detector-gated boot identity, safe-state, firmware identity, source/reference commit, and wrapper-owned serial-capture claims. It does not verify active voltage, fan, mining, self-test hardware, erase, rollback, recovery, or runtime display/input parity. |
| `power-telemetry` | `flash-command-evidence.json`, `flash-monitor.log`, `redaction-review.md`, optional read-only telemetry notes from an existing safe route | May conclude passed only for INA260 read-only current, bus voltage, power, freshness, read status, or explicit safe-unavailable telemetry. It cannot verify DS4432U writes, ASIC power sequencing, or stale cached values. |
| `voltage-control` | `flash-command-evidence.json`, `flash-monitor.log`, `redaction-review.md`, optional bounded-probe artifact from a later plan | Active-control packs may remain `hardware evidence pending` when bounded probes or recovery are absent. A passed conclusion for active DS4432U voltage control requires bounded input values, stop conditions, recovery steps, fail-closed output, logs, and redaction review. |
| `thermal-fan` | `flash-command-evidence.json`, `flash-monitor.log`, `redaction-review.md`, optional read-only temperature/RPM notes or later bounded fan artifact | Read-only thermal or RPM observations may pass for exact board-named claims. Fan duty actuation, overheat handling, fan-fault handling, and physical duty effects remain `hardware evidence pending` without bounded recovery. |
| `self-test-watchdog-load` | `flash-command-evidence.json`, `flash-monitor.log`, `redaction-review.md`, optional bounded liveness or responsiveness artifact from a later plan | Safe-unavailable self-test status, watchdog supervisor yield logs, and exact liveness observations may be cited narrowly. Self-test hardware submodes, fault injection, reboot behavior, ASIC work, and mining stress remain `hardware evidence pending` unless a later plan adds bounded recovery. |
| `display-input` | `flash-command-evidence.json`, `flash-monitor.log`, `redaction-review.md`, optional physical startup display observation notes | Startup SSD1306 observations may pass only as startup display evidence. Runtime display, screen flow, LVGL parity, and input behavior stay `hardware evidence pending` until a real runtime route is safely exercised. |
| `parity-guard` | `parity-report.txt` or copied `just parity` output summary, `redaction-review.md` if logs include captured terminal output | May conclude passed when `just parity` rejects unsafe safety-critical overclaims and reports no invalid checklist claims. It does not itself verify hardware behavior. |

## Generated By Repo Commands

`flash-command-evidence.json` and `flash-monitor.log` are generated only by:

```bash
just flash-monitor board=205 port=<port> evidence-dir=docs/parity/evidence/phase-11-safety-controller-hardware-regression-evidence
```

The command may run only after `just detect-ultra205` finds exactly one likely ESP USB serial port and its `espflash board-info --chip esp32s3 --port <port> --non-interactive` check succeeds.

`redaction-review.md` is written by the executor after inspecting captured artifacts. It must not be generated blindly from a template without reviewing the exact JSON, log, terminal, API, WebSocket, and manual-observation content being cited.

No generated JSON or log artifact should be created by hand. If a pack needs a new artifact type, a later plan must name the repo-owned command or bounded probe that writes it.

## Required Metadata

Every Phase 11 evidence pack must include:

- Board `205`.
- Selected `port=<path>` from the detector gate.
- Source commit.
- Reference commit.
- Exact command or probe.
- Package manifest or firmware identity when applicable.
- Logs or captured output used for the claim.
- Observed behavior.
- Conclusion scoped to the exact component claim.
- Secret-redaction review.

For read-only telemetry, also record freshness, read success or failure status, and whether the value is live, cached, unavailable, or explicitly pending.

For any later bounded active-control probe, also record input limits, abort conditions, recovery steps, fail-closed output, and the owner responsible for deciding whether the board returned to a safe state.

## Pending Evidence Semantics

`hardware evidence pending` is a valid conservative conclusion when detector, recovery, stimulus, bounded probe, redaction, or route prerequisites are absent.

Pending evidence must be used instead of broad success language for active voltage writes, fan duty actuation, overheat stimulus, fault injection, self-test hardware submodes that touch voltage/fan/ASIC work, mining stress, raw I2C writes, raw flash writes, erase, rollback, interrupted-update procedures, runtime display/input behavior, and watchdog/load stress.

Narrow safe-unavailable, read-only telemetry, startup display, or safe boot observations may be cited only as the exact observed subclaim. They must not verify the full checklist row when active subclaims remain pending.

## Do Not Hand Edit Generated Artifacts

Do not hand edit `flash-command-evidence.json` or `flash-monitor.log`. Regenerate them with the repo-owned command after detector success.

Do not paste secrets into evidence files. Do not commit pool credentials, Wi-Fi credentials, private endpoints, NVS secret values, API tokens, or unredacted terminal output.

Do not replace failed or untrusted wrapper evidence with raw monitor captures. If wrapper evidence is missing or untrusted, record the affected pack as `hardware evidence pending` and keep the checklist row below `verified`.
