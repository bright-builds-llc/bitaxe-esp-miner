# Phase 28.1.1.1 Research

## Summary

Phase 28.1.1.1 should not start by changing Rust mining logic. Phase 28.1.1 already established that the Rust firmware dispatches pool-derived work and gets BM1366 register-read proof, but it still records zero correlated nonce and zero accepted/rejected share markers. The highest-signal next step is a redaction-safe field comparator that turns local upstream scratch `SERIALTX_DEBUG` / `SERIALRX_DEBUG` captures and Rust `asic_uart_trace` captures into a committed field-level verdict.

## Existing Assets

### Upstream scratch capture

- `scripts/phase28.1.1-upstream-wire-capture.sh` verifies the pinned reference, creates or reuses `scratch/upstream-wire-capture/esp-miner`, and prints the SERIALTX/SERIALRX debug enablement and recovery commands.
- The script intentionally does not build or flash upstream. That step remains operator-local because upstream ESP-IDF C build details and local runtime inputs are outside the read-only reference contract.
- Raw upstream bytes should stay under `scratch/upstream-wire-capture/captures/`, which is gitignored.

### Rust hardware capture

- `scripts/phase28.1.1-wire-parity-capture.sh` runs the Phase 27 live hardware bridge package, flashes the Rust image, captures at least 360 seconds, and applies local pool input through `scripts/phase21-pool-input-bridge.sh` when a fresh monitor-derived device URL appears.
- `firmware/bitaxe/src/asic_adapter/uart.rs` emits `asic_uart_trace=tx len=... hex=...` and RX chunk/complete markers whenever the Phase 27 live hardware bridge mode or `BITAXE_ASIC_UART_TRACE=1` is active.
- Ignored hardware output belongs under `.planning/phases/**/hardware-runs/`; committed evidence should summarize marker counts and field verdicts only.

### Typed Rust frame surfaces

- `crates/bitaxe-asic/src/bm1366/work.rs` owns `Bm1366WorkFields`, `Bm1366WorkPayload`, and job frame construction. It already encodes `job_id`, `num_midstates`, starting nonce, `nbits`, `ntime`, merkle root, previous block hash, and version into an 82-byte payload and 88-byte job frame.
- `crates/bitaxe-stratum/src/v1/mining.rs` owns field construction from Stratum notify/extranonce data, including coinbase hashing, merkle root folding, previous block hash transforms, `nbits`, `ntime`, and version bytes.
- `crates/bitaxe-stratum/src/v1/live_runtime.rs` owns upstream-aligned first job id pre-increment, work regeneration, version rolling configure/startup, and submit classification.
- `crates/bitaxe-asic/src/bm1366/result.rs` distinguishes register-read proof from job nonce results; register-read success is chip-alive evidence, not hashing proof.

## Comparator Design

The comparator should parse two local inputs:

- Upstream SERIALTX/SERIALRX log text from ignored scratch captures.
- Rust `asic_uart_trace` log text from ignored hardware captures.

It should produce a committed, redaction-safe report with only field names, lengths, classifications, counts, and `match` / `mismatch` verdicts. The comparator must not copy raw byte streams into committed docs.

Minimum fields to compare:

| Area | Fields |
| --- | --- |
| Init sequence | chip-detect prelude, enumerate/read commands, static init write sequence, dynamic difficulty mask, PLL, HCN/nonce-space frame |
| Job frame envelope | preamble, header, total length, payload length, CRC validity category |
| Job payload | `job_id`, `num_midstates`, starting nonce, `nbits`, `ntime`, merkle root transform, previous block hash transform, version bytes |
| Sequencing | first production job id, +8 stride, clean-jobs invalidation, work regeneration order, dispatch-before-poll behavior |
| Version rolling | `mining.configure` presence, negotiated mask handling, base notify version bytes, version bits in nonce submission path |

Implementation options:

1. Add a small host Rust module or CLI under `tools/parity` that parses logs and emits a Markdown or JSON verdict.
2. Add a repo-owned script under `scripts/` if it stays deterministic and testable.

The Rust `tools/parity` route is preferable if the comparator needs structured tests and long-term reuse. A script is acceptable for a narrow phase helper if it has fixtures and shell syntax checks.
