## Implementation Decisions

### Evidence ladder
- **D-01:** Start from the Phase 28.1.1 facts: Rust dispatched pool-derived BM1366 jobs and parsed register-read proof, but recorded zero `asic_production_status=result_correlated`, zero accepted/rejected share markers, and flat/negative hashing-class power evidence.
- **D-02:** Treat upstream ESP-Miner v2.14.0 mining on the same Ultra 205 as board-health proof already established by Phase 28.1; this phase does not redo upstream A/B as a first diagnostic.
- **D-03:** The first discriminator is upstream scratch `SERIALTX_DEBUG` / `SERIALRX_DEBUG` job-byte capture plus a field-level comparator against Rust UART/job-frame evidence.

### Upstream capture safety
- **D-04:** Never modify `reference/esp-miner`; use only the gitignored `scratch/upstream-wire-capture/` copy prepared by `scripts/phase28.1.1-upstream-wire-capture.sh`.
- **D-05:** Raw upstream UART bytes, Rust UART bytes, pool payloads, endpoints, worker strings, owner addresses, device URLs, SSIDs, MACs, credentials, and NVS secret values remain local-only and must not be committed.
- **D-06:** Committed upstream evidence may record category labels, source/reference commits, command shapes, capture duration, and field-level verdicts, but not raw byte streams or secret-bearing runtime values.

### Comparator scope
- **D-07:** Compare field-level init and job construction, including job id sequence, `num_midstates`, starting nonce / nonce-space behavior, `nbits`, `ntime`, merkle root transform, previous block hash transform, version/version-rolling behavior, and dispatch order.
- **D-08:** Prefer a repo-owned parser/comparator or script over ad hoc manual string inspection. The output must be deterministic and redaction-safe so it can be committed as evidence.
- **D-09:** A comparator verdict of "match" is useful evidence only if it names the compared fields and source artifacts; a vague "looks similar" verdict is not enough.

### Patch policy
- **D-10:** Patch exactly one confirmed Rust divergence at a time, with a focused unit or fixture test in `bitaxe-asic` or `bitaxe-stratum`.
- **D-11:** Do not add another investigation flag or speculative fallback unless the comparator or hardware evidence names a concrete behavior difference that requires it.
- **D-12:** If upstream and Rust fields match but no nonce appears, move diagnosis below Stratum/job-frame construction: ASIC enable/power sequencing, reset timing, PLL/baud timing, post-init side effects, or missing upstream hardware-side sequencing.

### Hardware evidence and verification
- **D-13:** Every hardware run must pass `just detect-ultra205`, `just verify-reference`, use board `205`, and capture at least 360 seconds with a wall-clock budget above the capture timeout.
- **D-14:** Final verification is `passed` only when committed redacted evidence contains `asic_production_status=result_correlated` and at least one live `share_submission_status=accepted` or `share_submission_status=rejected` marker, or when the roadmap definition is narrowed by explicit user-approved scope. Without that, verification must be `gaps_found` or `blocked_safe_prerequisite`.
- **D-15:** Phase 30 checklist promotion remains deferred until this phase produces eligible redacted share-outcome evidence.

### the agent's Discretion
- The exact comparator file format, parser implementation language, and evidence doc layout are implementation choices, provided the artifacts are deterministic, redaction-safe, and easy to verify.
- The phase may stop with a narrower blocker if upstream scratch build or hardware prerequisites cannot be completed safely, but it must record the blocker precisely.
