- **D-04:** Add a redaction-safe **result-path comparator** (counts/order only) that makes upstream-vs-Rust RX/poll asymmetry first-class evidence before any firmware patch.
- **D-05:** Comparator must record at least: `chip_id_read_tx`, `result_read_attempt`, `rx_result_frame` / nonce-class RX, `register_read_parsed`, `result_correlated`, fake-pool submit observed (boolean), and a recommended investigation label. No raw UART bytes or credentials in committed output.
- **D-06:** Prefer extending or composing with `scripts/phase28.1.1.1-below-job-byte-sequence-compare.mjs` and the source-work fake-pool capture path rather than inventing a parallel capture stack.

### Hypothesis order
- **D-07:** Investigate in this order: (1) result acquisition model — upstream continuous `receive_work`/SERIAL_rx vs Rust aggressive `result_read_attempt` polling and whether Rust starves or mis-classifies nonce frames; (2) ASIC enable / power / reset sequencing side effects not in the current 12-event comparator; (3) missing upstream runtime side effect outside job-frame bytes (beyond the falsified 2s post-max-baud delay).
- **D-08:** Upstream reference behavior is passive RX via `receive_work` → `SERIAL_rx` in a dedicated result task (`asic_result_task.c`), not TX-driven result-register polling. Rust default continuous path already arms a listener and polls RX; the comparator must quantify whether Rust is still TX-probing, mis-parsing, or missing an enable/power side effect.

### Patch policy
- **D-09:** Patch exactly one confirmed Rust divergence at a time, with a focused unit/fixture test when the change touches pure crates.
- **D-10:** Do not add another investigation flag unless evidence names a concrete behavior difference that requires it (carry forward 28.1.1.1 D-11).
- **D-11:** Reuse Phase 27 package path and existing `--investigation` / `--chip-detect-investigation` wiring only when an A/B needs a named mode already supported by capture scripts.

### Hardware evidence and verification
- **D-12:** Every hardware run must pass `just detect-ultra205`, `just verify-reference`, use board `205`, and capture at least 360 seconds with wall-clock budget above the capture timeout.
- **D-13:** Prefer deterministic source-work fake-pool alignment (reuse `scripts/phase28.1.1.1-source-work-aligned-capture.sh` / fake-pool helpers) so submit presence is comparable without live-pool secrets.
- **D-14:** Final verification is `passed` only when committed redacted evidence contains `asic_production_status=result_correlated` and at least one share-submission marker (fake-pool or live accepted/rejected), or when the blocker is explicitly narrowed with redacted evidence and user-approved scope. Otherwise `gaps_found` or `blocked_safe_prerequisite`.
- **D-15:** Phase 30 checklist promotion remains deferred until eligible redacted share-outcome evidence exists.
