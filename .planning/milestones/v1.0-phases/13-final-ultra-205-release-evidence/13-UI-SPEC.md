# Phase 13 - UI Design Contract

> Visual and interaction contract for final Ultra 205 release-evidence surfaces. Generated by gsd-ui-researcher, verified by gsd-ui-checker.

***

## Design System

Phase 13 does not create a new frontend, replace AxeOS, rewrite Angular, or introduce a component registry. The design system for this phase is the existing Rust-owned static fallback and recovery route contract plus the response/status behavior needed to capture release evidence.

| Property | Value |
|----------|-------|
| Tool | none |
| Preset | not applicable |
| Component library | none for Phase 13; existing Rust-owned HTML/CSS only |
| Icon library | none |
| Font | Static fallback: system UI stack, weight 400. Recovery: `"courier new"`, monospace, weight 400 |

Sources: `.planning/phases/13-final-ultra-205-release-evidence/13-CONTEXT.md`, `.planning/phases/13-final-ultra-205-release-evidence/13-RESEARCH.md`, `.planning/phases/07-ota-filesystem-and-release-packaging/07-UI-SPEC.md`, `firmware/bitaxe/static/www/index.html`, `firmware/bitaxe/static/www/assets/app.css`, `firmware/bitaxe/static/recovery_page.html`, `crates/bitaxe-api/src/static_plan.rs`, and `crates/bitaxe-api/src/update_plan.rs`.

Hard constraints:

- Do not add shadcn, Tailwind, Radix, React, Next.js, Vite, Solid, Angular build work, or any third-party registry block.
- Do not redesign AxeOS, the fallback page, the recovery page, update flows, route layout, colors, typography, or controls.
- Do not introduce marketing, onboarding, dashboard, settings, or admin product UI.
- Do not change public route copy to make evidence easier to capture. Evidence helpers must record existing responses.
- Do not claim OTAWWW success. `/api/system/OTAWWW` remains the REL-03 gap with public response `Wrong API input` unless a later approved plan adds whole-`www` update, recovery, and interrupted-update hardware-regression evidence.
- Do not run destructive UI or route experiments outside a Phase 13 plan that names the command, recovery path, stop conditions, and evidence artifacts.

***

## Spacing Scale

Declared values (must be multiples of 4):

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px | Inline status gaps only if a touched existing surface needs them |
| sm | 8px | Existing fallback action gaps and recovery response gaps |
| md | 16px | Existing recovery form gap, fallback heading/body spacing, horizontal control padding |
| lg | 24px | Existing page padding, card padding, form top margin, response top margin |
| xl | 32px | Reserved for docs/evidence section spacing only; do not add to device pages |
| 2xl | 48px | Reserved for generated evidence reports only; do not add to device pages |
| 3xl | 64px | Not used in Phase 13 device UI |

Exceptions:

- The existing fallback and recovery pages use only the spacing they already declare: `8px`, `16px`, and `24px`, plus `44px` minimum control height.
- File inputs, links styled as buttons, and buttons must keep a minimum hit target height of `44px`.
- Recovery page buttons keep square corners (`border-radius: 0`) and the fallback action links keep `8px` radius. Do not normalize one to the other.
- Missing static redirect, OTA, OTAWWW, and API coexistence checks have no visual layout; evidence must record HTTP status, headers, body snippets, and conclusions instead of adding a rendered UI.

***

## Typography

Use only two font weights: `400` and `500`. Do not add display typography or hero-scale copy.

| Role | Size | Weight | Line Height |
|------|------|--------|-------------|
| Body | 14px | 400 | 1.5 |
| Label / Status | 14px | 400 | 1.5 |
| Evidence Helper Subheading | 16px | 400 | 1.5 |
| Heading | 28px | 500 | 1.2 |

Typography rules:

- Existing device pages must remain visually unchanged unless a Phase 13 bug fix directly touches them.
- `16px` is reserved only for any new local evidence helper HTML/reporting view. It must not be introduced into `firmware/bitaxe/static/www/index.html` or `firmware/bitaxe/static/recovery_page.html` without a documented bug fix.
- Public status strings must stay short and exact: `uploading...`, `Upload successful!`, `Error uploading!`, `Request failed`, `Starting...`, `Working ({percent}%)`, `Validation Error`, and `Rebooting...`.
- Do not use color alone for state. The state text must carry the outcome.

***

## Color

Preserve the existing dark surfaces. Do not introduce a new palette.

| Role | Value | Usage |
|------|-------|-------|
| Dominant (60%) | `#070d17` | Static fallback page background |
| Secondary (30%) | `#111923`, `#2e3744`, `#a8b7c4` | Fallback status panel, border, and subdued eyebrow text |
| Accent (10%) | `#f80421` | Fallback action link border and hover/focus fill only |
| Destructive | `#f48fb1` | Destructive/error affordances only if a new evidence helper view is added; existing device pages use text states instead |

Accent reserved for: `Open recovery` link border/focus/hover, `Release metadata` link border/focus/hover, and any future evidence helper primary action approved by a Phase 13 plan. Accent must not be used for ordinary route names, evidence filenames, warnings, or all interactive text.

Recovery color exception:

| Role | Value | Usage |
|------|-------|-------|
| Recovery dominant | `#000000` | Recovery page background and button background |
| Recovery text/accent | `#00ff00` | Recovery page text, button border, and hover/focus inversion |

Accessibility rules:

- Fallback action links and recovery buttons must retain visible hover and focus states.
- The recovery status and response outputs must remain `aria-live="polite"`.
- The fallback page must keep `main` labelled by `#page-title`.
- Evidence screenshots, if captured, must show readable copy at mobile width and desktop width; no text may overlap or truncate.

## Phase 13 Static, Recovery, And Evidence Contract

### Route Interaction Contract

| Surface | Required behavior | Evidence requirement |
|---------|-------------------|----------------------|
| `GET /` | Serve the existing Rust-owned fallback HTML. Body must include `AxeOS unavailable`, `Open recovery`, and `Release metadata`. | Record method, URL, status, `Content-Type`, body markers, sanitized snippet, and conclusion. |
| `GET /assets/app.css.gz` | Serve the packaged gzip CSS asset. Direct `.gz` request may omit `Content-Encoding`; `GET /assets/app.css` may serve the gzip variant with `Content-Encoding: gzip`. | Record status, `Content-Type: text/css`, `Cache-Control: max-age=2592000`, `Content-Encoding` if present, response size or checksum, and whether decompressed bytes match `firmware/bitaxe/static/www/assets/app.css`. |
| Missing static path | Redirect to `/` with `302 Temporary Redirect` and body `Redirect to the captive portal`. | Record status, `Location`, content type, body, redirect-follow policy, and final conclusion. |
| `GET /recovery` | Serve embedded recovery HTML with `AxeOS Recovery`, `www.bin`, `Upload`, `Restart`, and `Response:`. Must remain available when SPIFFS/static files are unavailable. | Record status, `Content-Type: text/html`, body markers, and whether the route remains reachable after any recovery/destructive step. |
| `/api/*` coexistence | API routes must not be captured by static wildcard routing. Unknown `/api/*` returns the JSON 404 body. | Probe at least one known API route and one unknown `/api/*` route, or record why API probing is blocked. |
| `GET /api/ws` and `GET /api/ws/live` | WebSocket routes must remain registered separately from static wildcard. | Evidence may record route registration plus bounded upgrade attempt; do not require a visual UI. |
| `POST /api/system/OTA` | Valid firmware upload uses `esp-miner.bin`, reports public success or error text, schedules reboot only after success, and requires post-reboot identity/boot-validation evidence before parity claims. | Record artifact checksum, status/body, update status labels, reboot observation, post-reboot firmware commit, partition/boot-validation log lines, and recovery action if needed. |
| `POST /api/system/OTAWWW` | Remains fail-closed with public response `Wrong API input` unless full whole-`www` update evidence is added by plan. | Record status/body and log gap owner. Do not mark OTAWWW verified from `www.bin` package generation or recovery page upload UI alone. |

### Evidence Capture Contract

Every live HTTP/static/recovery/OTA evidence row must record:

| Field | Required value |
|-------|----------------|
| Board | `205` / Ultra 205 |
| Source identity | Final source commit from the package manifest |
| Reference identity | Pinned `reference/esp-miner` commit from the package manifest |
| Package manifest | Path to `bazel-bin/firmware/bitaxe/bitaxe-ultra205-package.json` or copied manifest evidence |
| Device target | `DEVICE_URL` source, sanitized when private |
| Request | Method, path, expected status/body marker, actual status/body marker, selected headers |
| Route conclusion | `passed`, `blocked`, or `pending`; never `verified` unless the checklist evidence class supports it |
| Redaction | Explicit review result before commit |

If `DEVICE_URL` is missing, ambiguous, unreachable, or targets the wrong device, record `DEVICE_URL status: blocked` and keep live HTTP/static/recovery/OTA rows below `verified`.

### Recovery And Destructive Interaction Contract

| Action | UI/interaction rule | Evidence boundary |
|--------|---------------------|-------------------|
| Recovery page load | `/recovery` is the operator recovery surface. Do not add alternate recovery UI. | Page load alone proves reachability, not successful static partition recovery. |
| Recovery upload | Existing form posts selected bytes to `/api/system/OTAWWW` and displays response text. | Because OTAWWW is a gap, a failed `Wrong API input` response is expected unless a later plan changes the route with hardware-regression evidence. |
| Recovery restart | Existing `Restart` button posts to `/api/system/restart` and displays response text. | Restart must be captured with post-action static/API reachability before claiming recovery success. |
| Firmware OTA | Existing route response is not rollback proof. | Rollback requires bootloader or boot-validation evidence from the post-update state. |
| Large erase | No in-app UI. Plan/runbook gated command only. | Must capture factory image recovery path, post-erase boot, filesystem, `/recovery`, and API reachability before any verified claim. |
| Failed/interrupted update | No ad hoc UI changes. Plan/runbook gated route or command only. | Must capture failure point, public/internal status, recovery steps, final operability, and redaction review. |

### Non-Claims

- `/assets/app.css.gz` package presence is not live static evidence unless the response is captured from the just-flashed device.
- Recovery page reachability is not OTAWWW parity.
- Invalid upload rejection is not rollback proof.
- Serial route registration is not live HTTP proof.
- `www.bin` generation is not whole-`www` update proof.
- Any blocked evidence must remain blocked or pending in docs/checklist; do not convert blocked copy into verified release language.
