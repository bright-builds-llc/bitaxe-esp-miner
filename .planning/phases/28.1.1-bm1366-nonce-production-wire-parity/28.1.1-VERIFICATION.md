## Evidence-Correctness Gaps

### 1. Physical lifecycle and phase goal remain incomplete

The exact five-stage reinit member was never obtained, the watcher never armed, no both-power-path cold start occurred, and no cold-start member exists. Separately, there is still no correlated BM1366 result, hashing-class power rise, or live accepted/rejected share.

### 2. Five-second total-power removal is instructed but not enforced

`scripts/phase28.1.1-accepted-state-diagnostic.sh` returns on the first absent-port observation and immediately waits for reappearance. The success fixture uses `present absent present`. A brief disconnect can therefore satisfy the workflow even though Plan 10 requires both power paths to remain removed for at least five seconds.

### 3. Latest permitted reattach can miss the final replay

Replay occurs at approximately 0, 2, ..., 88 seconds and expires at 90 seconds, while the wrapper permits reattachment through 89 seconds. A monitor attached after the 88-second replay may observe no later replay. The current boundary test jumps directly to the final millisecond and does not consume the preceding cadence, so it does not test the real schedule.

### 4. Standalone accepted-state classifiers overclaim mutually missing stages

The Rust classifier and `phase28.1.1-accepted-state-compare.mjs` skip a stage when both sides lack it. Empty or matching partial inputs can therefore become `match` instead of `unavailable`. The stricter lifecycle comparator protects its own path, but the pure API and standalone comparator remain incorrect for partial or empty input.

### 5. The Plan 10 denylist command is malformed

The literal Task 3 `rg` expression contains an unclosed character class. Because it is prefixed with `!`, the regex parse error can be converted into a successful shell status. A corrected equivalent query found no prohibited content in the promoted artifacts, but the automated gate itself does not currently prove that property.
