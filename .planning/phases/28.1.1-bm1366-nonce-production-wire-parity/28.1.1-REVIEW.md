## Warnings

### WR-01: Lifecycle capture does not enforce the required five-second total-power-off interval

**File:** `scripts/phase28.1.1-accepted-state-diagnostic.sh:327-334`

**Issue:** The wrapper tells the operator to remove both power paths and wait five seconds, but it accepts the first observed `absent` result and immediately starts waiting for `present`. There is no elapsed-time or consecutive-absence check. The success fixture at `scripts/phase28.1.1-accepted-state-diagnostic-test.sh:220` explicitly passes with `present absent present`, so a sub-second USB disappearance can satisfy the workflow. A subsequent boot marker proves a reboot occurred, but it does not prove the plan's required five-second full-power removal. The resulting evidence could therefore claim the specified physical cold-start procedure without measuring its defining interval.

**Fix:** After arming, require the selected USB port to remain absent for at least five measured seconds before accepting reappearance. Fail closed if it returns early, and update the fixture to provide at least five consecutive absent observations plus tests for early return and the exact boundary. Keep the shared reattach budget explicit after accounting for this minimum absence period.

### WR-02: The accepted reattach timeout can outlive the final periodic replay

**Files:** `crates/bitaxe-api/src/logs.rs:18-21`, `crates/bitaxe-api/src/logs.rs:246-265`, `scripts/phase28.1.1-accepted-state-diagnostic.sh:64-65`

**Issue:** The firmware replay window is 90 seconds with a two-second cadence, while the wrapper accepts a reattach timeout of up to 89 seconds. Under normal repeated calls, replay is due near 0, 2, ..., 88 seconds and is exhausted at 90 seconds. If USB reappears after the 88-second replay but within the allowed 89-second timeout, monitor startup occurs after the last marker and the capture cannot observe another replay. The boundary unit test calls `take_due` for the first time at the last millisecond, so it does not cover the actual repeated-cadence schedule and misses this gap.

**Fix:** Give monitor attachment and at least one subsequent cadence a documented margin—for example, extend the replay window beyond the maximum reattach budget or lower the maximum timeout—and add a test that consumes every earlier cadence before asserting a replay remains due after the latest permitted attachment time. Include monitor process startup in the margin rather than relying only on `timeout < window`.

### WR-03: Missing stages on both sides are classified as a match

**Files:** `crates/bitaxe-asic/src/bm1366/accepted_state.rs:169-205`, `scripts/phase28.1.1-accepted-state-compare.mjs:146-186`

**Issue:** Both classifiers skip a stage when upstream and Rust observations are absent. Consequently, empty inputs—and matching inputs that contain only a subset of the five stages—fall through to `match`. This contradicts the classifier contract that missing safe-readable observations route to unavailable/lifecycle investigation and creates an overclaim path if the comparator is invoked outside the lifecycle wrapper's stricter exact-stage parser. The current wrapper usually protects absent whole logs by synthesizing unavailable markers, but the pure API and standalone comparator remain incorrect for partial or empty inputs.

**Fix:** Treat every absent expected stage as a missing observation, or explicitly require the five-stage set before comparison. Return `Unavailable` / `unavailable` when either side lacks any required stage, and add focused Rust and Node tests for empty inputs, matching three-of-five inputs, and one-sided missing stages.
