<threat_model>
## Trust Boundaries

| Boundary | Description |
| --- | --- |
| Operator-supplied `DEVICE_URL` to telemetry helper | The URL determines which device receives API/WebSocket probes. |
| Device API responses to committed evidence | Responses can contain private network data or stale telemetry. |
| WebSocket route status to frame/cadence claim | Route presence can be mistaken for live frame proof. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
| --- | --- | --- | --- | --- |
| T-14-05-01 | Spoofing | `DEVICE_URL` | mitigate | Require explicit URL, no network inference, sanitized target recording, and package/detector context before passing evidence. |
| T-14-05-02 | Information Disclosure | API bodies and telemetry logs | mitigate | Sanitize credentials, private endpoints, NVS values, API tokens, IPs, and MACs before writing committed summaries. |
| T-14-05-03 | Tampering | telemetry conclusions | mitigate | Record status codes, headers, sanitized snippets, and explicit blocked/pending statuses instead of hand-written pass claims. |
| T-14-05-04 | Repudiation | WebSocket frame proof | mitigate | Record frame proof as pending when no maintained client exists; do not hand-roll WebSocket framing. |
</threat_model>
