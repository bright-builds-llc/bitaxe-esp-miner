## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Raw user/API/NVS values -> validation domain types | Untrusted primitive values cross into the config core. |
| REST field names -> NVS write decisions | Future API handlers will rely on this mapping to avoid duplicate validation logic. |
| Hardware-sensitive settings -> later hardware control | Frequency, voltage, and fan settings must stay inert until hardware evidence exists. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |

| T-02-01 | Tampering / Elevation of Privilege | `validation.rs` and `settings.rs` | mitigate | Parse frequency, voltage, fan, temp, hostname, port, TLS, bool, and protocol values into bounded domain types; accepted decisions return inert `NvsWrite` data only. |
| T-02-02 | Information Disclosure | `settings-updates.json` | mitigate | Fixture cases use public upstream pool defaults and synthetic invalid values only; no Wi-Fi credentials or private operator pool credentials. |
| T-02-03 | Tampering / Denial of Service | REST-to-NVS mapping in `settings.rs` | mitigate | Use Plan 02 schema rows for exact NVS keys and include tests for active plus legacy mirror writes. |
| T-02-04 | Denial of Service | Float update handling | mitigate | Frequency updates serialize through `FloatString`; Plan 02 corrupt-load fallback remains the read-side guard. |
| T-02-05 | Spoofing / Safety governance failure | Board validation | mitigate | `BoardVersion` refuses active hardware-verified scope for every board except `205`; tests cover `601` as not hardware-verified. |
