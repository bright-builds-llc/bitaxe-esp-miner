<supported-image-policy>
The implementation must cite and encode the following concrete policy from pinned local sources, not from memory or a network lookup:

1. Layout and general limits come from `.embuild/espressif/esp-idf/v5.5.4/components/bootloader_support/include/esp_app_format.h` and `.embuild/espressif/esp-idf/v5.5.4/components/bootloader_support/src/esp_image_format.c`: 24-byte v2 image header, 8-byte segment headers, magic `0xe9`, `ESP_IMAGE_MAX_SEGMENTS == 16`, every data length divisible by four and below 16 MiB, XOR checksum over segment data, exact padding/trailer, and appended SHA-256. Pinned `verify_segment_header` accepts an aligned zero data length and `verify_load_addresses` explicitly treats a zero-length segment as valid; the host validator must do the same. The canonical application profile still requires at least one segment because the application descriptor and executable entry must be present, but it must not reject an additional zero-length segment merely for being empty or subject its meaningless address to non-empty destination containment. An empty segment cannot contain the entry point or create a destination range; if its address classifies as mapped DROM/IROM, pinned mapped-offset congruence still applies.
2. The only supported package-producing command is the current `scripts/package-firmware.sh` `esptool --chip esp32s3 elf2image --version 2 --flash_size 16MB --flash_mode dio --flash_freq 80m --elf-sha256-offset 0xb0 --min-rev-full 0 --max-rev-full 99`. Therefore header admission is exact for SPI mode `2` (DIO), speed/size byte `0x4f` (80 MHz/16 MiB), WP pin `0xee`, zero SPI drive settings, chip ID `9`, minimum revision byte/full `0`, maximum revision full `99`, four zero reserved bytes, and `hash_appended == 1`. Other syntactically valid ESP image variants are unsupported and rejected.
3. The broad address families come from `.embuild/espressif/esp-idf/v5.5.4/components/soc/esp32s3/include/soc/soc.h`: DROM `[0x3c000000, 0x3e000000)`, DRAM `[0x3fc88000, 0x3fd00000)`, IRAM `[0x40370000, 0x403e0000)`, IROM `[0x42000000, 0x44000000)`, RTC slow/data `[0x50000000, 0x50002000)`, and RTC fast `[0x600fe000, 0x60100000)`. For non-empty loaded DRAM/IRAM, broad family containment is insufficient: pinned `verify_load_addresses` also rejects overlap with the runtime bootloader stack, `_dram_start.._dram_end`, `_loader_text_start.._loader_text_end`, and the D/IRAM aliases of those regions. Exact linker-symbol ends and the current stack pointer are build/runtime values and schema-v3 artifacts contain neither a bootloader map nor those values, so host admission must not claim loader-complete bootability or pretend to reproduce those dynamic checks.
4. Instead, define an explicitly stricter canonical-producer envelope from pinned stable linker inputs. `.embuild/espressif/esp-idf/v5.5.4/components/esp_system/ld/esp32s3/memory.ld.in` fixes `SRAM_IRAM_END = 0x403cb700` and derives `SRAM_DRAM_END = 0x3fcdb700` with the ESP32-S3 D/IRAM alias offset `0x006f0000`; `.embuild/espressif/esp-idf/v5.5.4/components/bootloader/subproject/main/ld/esp32s3/bootloader.ld` fixes `iram_loader_seg` at `[0x403cb700, 0x403d2700)` and `dram_seg` at `[0x3fce2700, 0x3fce7700)` ahead of stack/ROM reservations. Admit non-empty IRAM only in `[0x40370000, 0x403cb700)` and non-empty DRAM only in `[0x3fc88000, 0x3fcdb700)`; retain the broad DROM, IROM, RTC slow/data, and RTC fast bounds above. This conservative producer envelope excludes the complete fixed loader region, bootloader data/stack area, and their D/IRAM aliases without depending on unavailable runtime symbols. Recomputed-envelope tests must use exact representative starts: IRAM loader `0x403cb700` and its DRAM alias `0x3fcdb700`; direct bootloader DRAM `0x3fce2700` and its IRAM alias `0x403d2700`; stack/ROM-reserved DRAM `0x3fce7710` and its IRAM alias `0x403d7710`. Every non-empty `load_addr.checked_add(data_len)` must succeed and its whole half-open range must lie in one admitted envelope; reserved low addresses such as `0` and `4`, gaps, peripherals, cross-boundary ranges, and wraparound are rejected.
5. ESP32-S3 does not define `SOC_MMU_PAGE_SIZE_CONFIGURABLE`, so the pinned `esp_image_format.c` branch sets mapping authority from `SPI_FLASH_MMU_PAGE_SIZE`, which `.embuild/espressif/esp-idf/v5.5.4/components/spi_flash/include/spi_flash_mmap.h` defines as `CONFIG_MMU_PAGE_SIZE`. The canonical current generated sdkconfig has `CONFIG_MMU_PAGE_SIZE=0x10000`; therefore DROM/IROM require `(segment_data_file_offset % 65536) == (load_addr % 65536)`. Separately, `.embuild/espressif/esp-idf/v5.5.4/components/esp_app_format/include/esp_app_desc.h` defines `esp_app_desc_t.mmu_page_size` as the MMU page size in log-base-2 format, and `.embuild/espressif/esp-idf/v5.5.4/components/esp_app_format/esp_app_desc.c` initializes it during the ESP-IDF application build as `31 - __builtin_clz(CONFIG_MMU_PAGE_SIZE)`. With `CONFIG_MMU_PAGE_SIZE=0x10000`, that formula yields descriptor value `16`; `elf2image` carries the descriptor from the ELF into the image rather than producing that value. Validate `mmu_page_size == 16` as exact current ESP-IDF application-build producer identity copied through `elf2image`, never as the authority that selects or independently proves the fixed ESP32-S3 64 KiB mapping rule.
6. The application entry point must be four-byte aligned and contained in the half-open range of at least one admitted non-empty IRAM or IROM segment. The `.embuild/espressif/esp-idf/v5.5.4/components/bootloader_support/src/bootloader_utility.c` transfer to `entry_addr` is the reason this is an executable invariant, not descriptive metadata.
7. The current canonical image, inspected with the managed pinned ESP-IDF/esptool tooling after `just package`, is the positive compatibility fixture: six segments across DROM, DRAM, IRAM, IROM, IRAM, and RTC data, with its entry inside IRAM, all loaded segments below the conservative producer ceilings, and 64 KiB-mapped offsets. The current generated sdkconfig and bootloader map selected by `scripts/package-firmware.sh` are verification evidence for the pinned constants, not new manifest inputs or runtime admission authorities. Tests must derive fixtures from or accurately mirror this concrete structure without committing generated binaries.

This policy closes the verifier's reproduced SYS-02 admission gaps for the exact current package producer while making a narrower claim than the ESP-IDF bootloader: it proves conformance to the canonical producer envelope and rejects known loader-reserved overlaps conservatively; it does not prove every device-time bootability condition that depends on unmanifested linker symbols, current stack state, eFuses, or hardware.
</supported-image-policy>
