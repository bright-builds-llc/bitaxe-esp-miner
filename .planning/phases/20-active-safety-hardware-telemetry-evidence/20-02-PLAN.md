<threat_model>
## Trust Boundaries

| Boundary | Description |
| --- | --- |
| USB serial -> repo evidence | Board-info and monitor output can contain unique hardware and network identifiers. |
| package manifest -> hardware evidence | A stale package can make later hardware claims refer to the wrong source commit. |
| serial log -> target selection | Redacted serial output must not become implicit `DEVICE_URL` evidence. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
| --- | --- | --- | --- | --- |
| T-20-02-01 | Spoofing | `just detect-ultra205` gate | mitigate | Continue only after exactly one likely ESP32-S3 port and board-info pass for board `205`; otherwise write blocked evidence. |
| T-20-02-02 | Repudiation / Integrity | Package-release evidence | mitigate | Compare package `source_commit` to `git rev-parse HEAD` and require release-gate output before later plans cite the package. |
| T-20-02-03 | Information Disclosure | `safe-baseline/*` artifacts | mitigate | Use `redact-evidence=true`, redact detector MAC/IP/SSID/URL values, and keep credential files unread. |
| T-20-02-04 | Spoofing | `target-lock.json` | mitigate | Disable network scans and reject target inference from committed redacted logs. |
| T-20-02-05 | Denial of Service | Hardware flash/monitor | mitigate | Run only repo-owned `just flash-monitor`; no raw active hardware, erase, recovery, or stress commands. |
</threat_model>
