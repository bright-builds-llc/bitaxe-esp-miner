Once-guards and baseline reset on session invalidation.
- All markers are fixed strings plus integer mW values only — no frame hex, register values, endpoints, or credential material (T-28.1-10).

**1. [Rule 3 - Blocking] Pre-existing corrupt esp-idf-sys ldflags broke `just build` at HEAD**
- **Found during:** Task 1 verification
- **Issue:** `just build` failed with `xtensa-esp-elf-g++: error: too many @-files encountered` while linking the ESP-IDF C bootstrap. Root cause: the generated response file `target/xtensa-esp32s3-espidf/release/build/esp-idf-sys-2a443813359d6cae/out/build/toolchain/ldflags` contained a self-reference (`@".../toolchain/ldflags"` as its own first line), producing infinite @-file recursion in GCC 14.2. Reproduced at clean HEAD (`git stash` + `just build`), so pre-existing stale build state — Plan 02 had verified via Bazel action cache without a cold esp-idf-sys rebuild.
- **Fix:** Deleted the corrupt `esp-idf-sys-2a443813359d6cae` build directory (repo `target/`; the Bazel execroot `target` is a symlink to it); embuild regenerated a clean ldflags and the build went green.
- **Files modified:** none (local generated build state only; nothing committed)
- **Verification:** `just build` green on both task commits; regenerated ldflags starts with `-nostartfiles`, no self-reference

**2. [Rule 3 - Blocking] `probe_register_read_tx` added to `asic_adapter/production.rs` (file not in plan's files_modified list)**
