## Warnings

### WR-01: Safe function transmutes caller-chosen lifetime to `'static` (latent unsoundness)

**File:** `firmware/bitaxe/src/safety_adapter/power_probe.rs:23-27`
**Issue:** `store_power_probe_bus(bus: BitaxeI2cBus<'_>)` is a safe `pub` function that does `unsafe { std::mem::transmute(bus) }` to `BitaxeI2cBus<'static>`. It is sound at the single current call site because `run_phase27_hardware_bring_up` bounds `I2C: I2c + 'static, SDA/SCL: ... + 'static` (`phase27_bring_up.rs:78-90`), so the concrete driver already owns `'static` hardware. But the signature accepts *any* lifetime: a future caller could pass a bus borrowing stack data and get a use-after-free with no compiler error. The `SAFETY` comment documents an invariant the type system could enforce directly.
**Fix:** The `unsafe` block is unnecessary. Tighten the signature and delete the transmute:
```rust
pub fn store_power_probe_bus(bus: BitaxeI2cBus<'static>) -> bool {
    let Ok(mut state) = probe_state().lock() else {
        return false;
    };
    if state.maybe_bus.is_some() {
        return false;
    }
    state.maybe_bus = Some(bus);
    true
}
```
The bring-up call site already satisfies `'static` because `BitaxeI2cBus::new` moves the `'static`-bounded peripherals into `I2cDriver`; annotate the local as `let mut bus: BitaxeI2cBus<'static> = BitaxeI2cBus::new(i2c, sda, scl)?;` if inference needs help.

### WR-02: `Regenerate` error arm can starve result polling until the next pool notify

**File:** `firmware/bitaxe/src/live_stratum_runtime.rs:785-792`
**Issue:** When `BridgeStep::Regenerate` fires and `runtime.regenerate_work()` returns `Err`, the arm returns `NoOp` but leaves the orchestrator's `maybe_last_dispatch_at` set. `next_step` then returns `Regenerate` on every subsequent iteration (elapsed only grows), so the bridge never reaches `Poll` again — nonce polling stops and each pump iteration burns a no-op ASIC step. This state is reachable: the runtime's regeneration context can be cleared without the pump seeing a `WorkInvalidated` event — e.g., a clean-jobs notify calls `invalidate_for_clean_jobs` (`live_runtime.rs:432`) and then fails work-building, and the resulting `Err` is swallowed by `handle_socket_line`'s pre-existing `.ok().flatten()` (`live_stratum_runtime.rs:1114`). The pump recovers on the next good notify (pools typically notify every ~30 s), but shares from already-dispatched work are lost in the window, and the `asic_bridge=timeout_streak` telemetry goes silent, which could mislead hardware-evidence interpretation.
**Fix:** Mirror the "nothing dispatchable" arm in `dispatch_production_work` — on regeneration failure, drop the cadence so the pump idles cleanly until fresh pool work:
```rust
Err(_) => {
    // Runtime lost its regeneration context (e.g. swallowed invalidation);
    // idle the cadence instead of spinning Regenerate forever.
    bridge.orchestrator.invalidate_session();
    AsicBridgeStepOutcome::NoOp
}
```
Add a host test in `bridge_orchestration.rs`/pump tests encoding "regeneration unavailable must not permanently shadow Poll".

### IN-02: `register_read_silent` marker also emitted for local TX faults

**File:** `firmware/bitaxe/src/asic_adapter.rs:375,381,386`
**Issue:** The post-init probe logs `asic_probe=register_read_silent stage=post_init` when frame encoding or UART TX fails locally — before any frame reached the chip. "Silent" is the discriminating signal for "chip did not respond"; conflating it with local TX faults could misclassify a firmware-side fault as chip silence during evidence review. The adjacent `register_read_frame_error`/`register_read_tx_error` warn lines disambiguate, but only if the reader correlates them.
**Fix:** Emit a distinct marker (e.g., `asic_probe=register_read_tx_failed stage=post_init`) on the encode/TX error paths and reserve `register_read_silent` for the read-timeout path (`asic_adapter.rs:404-407`).

### IN-03: Control mode still runs the orchestrator cadence as busy no-ops

**File:** `firmware/bitaxe/src/live_stratum_runtime.rs:709-716,787`
**Issue:** With `single_dispatch_bounded_read`, `note_dispatched` still arms the cadence, so 2 s after any dispatch `needs_step()` returns true on every iteration and each step maps `Regenerate` → `NoOp`. Harmless, but it publishes `StepKind::Asic` watchdog checkpoints and burns cycles that the pre-28.1 behavior being A/B'd did not, slightly reducing control-run fidelity.
**Fix:** In `needs_step`/`run_asic_bridge_step`, treat `Regenerate` as `Idle` when `!self.continuous` (e.g., have `needs_step` ignore `Regenerate` in control mode).

### IN-04: `regenerate_work` consumes counter and job id before enqueue can fail

**File:** `crates/bitaxe-stratum/src/v1/live_runtime.rs:306-328`
**Issue:** `extranonce2_counter += 1` and `next_asic_job_id()` are consumed before `enqueue_pool_work` (and `build`) can return `Err`, leaving gaps in the sequences on failure. Uniqueness is preserved (gaps are safe), so this is cosmetic; noting it so a future reader does not "fix" it by reusing values, which would be unsafe.
**Fix:** Optional — compute `let next_counter = self.extranonce2_counter + 1;` and commit `self.extranonce2_counter = next_counter` only after successful enqueue.
