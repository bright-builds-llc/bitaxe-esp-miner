### Pattern: pure decision + firmware interpreter

The repo's established shape (e.g., `Bm1366InitPlan` decisions → `interpret_action` in `asic_adapter.rs`; `GuardedMiningLoopInputs::plan()` → executor): pure crate computes a typed step/action, firmware executes it. `BridgeOrchestrator::next_step()` should return a typed step enum (e.g., dispatch / regenerate / poll(slice_ms) / idle) that the firmware pump interprets — this keeps the J2c-priority logic host-testable and the pump thin.

### Pattern: OnceLock static peripheral retention

`production.rs:28-70` is the sanctioned way to keep ESP-IDF peripherals alive across the boot-gate/bridge boundary (OnceLock<Mutex<State>> + transmute to `'static` with a SAFETY note). Reuse for the I2C bus if the power-delta probe samples mid-bridge.

- **Growing `live_stratum_runtime.rs` (2002 lines) with orchestration logic** — delegate; the file is already past the 628-line refactor trigger.
- **Second RX thread** — explicitly deferred; the ESP-IDF UART driver ring buffers RX between ≤100 ms poll slices.

## Common Pitfalls

### Pitfall 1: UART mutex sharing between dispatch and poll
**What goes wrong:** `ProductionAsicExecutor::execute` (dispatch) and `try_read_production_result` (poll) both lock the same `PRODUCTION_HANDLE` mutex (`production.rs:101,140`); a poll holding the lock for a full `poll_timeout_ms` blocks dispatch.
**Why it happens:** single retained UART behind one mutex; poll slice length = lock hold time.
**How to avoid:** keep poll slices ≤ `SOCKET_TIMEOUT_MS` (100 ms) as today (`live_stratum_runtime.rs:879-883`); orchestrator must never request a poll longer than one slice; dispatch-before-poll ordering inside one thread makes contention structural rather than timing-dependent.
**Warning signs:** `work_dispatched` markers spaced >> 2 s; watchdog checkpoint gaps.

### Pitfall 2: breaking first-dispatch extranonce2=0 golden parity
**What goes wrong:** starting the counter at 1 or incrementing before first build changes TX bytes vs checked-in golden fixtures.
**How to avoid:** counter starts 0; first `apply_notify` build uses 0 (current behavior — builder default); only `regenerate_work()` increments. Regression: existing `production_work.rs:590` fixture assert plus a new counter-sequence test.
**Warning signs:** any fixture test failure in `bitaxe-stratum` / `bitaxe-asic` goldens.

### Pitfall 3: firmware tests that never run
**What goes wrong:** placing the J2c regression test in `firmware/bitaxe` where `cargo test` cannot compile it (esp-idf-sys host gate) and Bazel has no test target — verification theater.
**How to avoid:** all executable tests in `crates/*`; treat firmware `#[cfg(test)]` as compile-contract only. `[VERIFIED: 26-03-SUMMARY.md:106]`

### Pitfall 4: Bazel srcs drift for the new module
**What goes wrong:** `bridge_orchestration.rs` added to Cargo module tree but not to the explicit `srcs` in `crates/bitaxe-stratum/BUILD.bazel` → `cargo test` passes, `just test` fails (or silently skips).
**How to avoid:** update `v1.rs` + `BUILD.bazel` in the same task; run both `cargo test --all-features` and `just test` in verification.

### Pitfall 5: evidence wrapper background-capture failure
**What goes wrong:** `phase27-live-hardware-bridge-evidence.sh` background `just flash-monitor` exited immediately in J2/J2b (empty capture dirs).
**How to avoid:** foreground `just flash-monitor board=205 port=<detected> capture-timeout-seconds=360 evidence-dir=<path>` per the J2c workaround; wall clock ≥ 420 s; do not treat early exit as failure before the full timeout. Also note `--duration-seconds` hard cap of 600 in the wrapper.

### Pitfall 6: INA260 probe with no bus
**What goes wrong:** planning the power-delta probe as a small logging change when the I2C bus is dropped at boot (`phase27_bring_up.rs:161`).
**How to avoid:** plan an explicit bus-retention task (OnceLock pattern) or sample via a repo-approved alternative; treat this as real firmware surface with its own verification.

### Pitfall 7: GSD frontmatter corruption
**What goes wrong:** standalone `---` in plan/summary/evidence Markdown bodies breaks the GSD frontmatter parser.
**How to avoid:** `***` or headings for body breaks in every artifact this phase writes (plans, run docs, defer notes).

### Pitfall 8: GPL expression leakage in new orchestration code
**What goes wrong:** porting `create_jobs_task.c` control flow verbatim into MIT `bridge_orchestration.rs`.
**How to avoid:** breadcrumb comments citing file:line (pattern: `ina260.rs:3`), independent Rust design (typed step enum, injected clock — already structurally different); no upstream identifiers or comment text.

### Pitfall 9: overclaiming in checklist promotion
**What goes wrong:** promoting STR-09/CFG-07 from orchestration-fix evidence without an accepted/rejected share.
**How to avoid:** `just parity --fail-on-invalid-verified` enforces this (`main.rs:1194-1240`); multiple `work_dispatched` markers falsify J2c but do NOT satisfy the share tier — keep tiers separate in run docs exactly as `phase27-04-defer-note.md` does.

### Pitfall 10: A/B flash without documented recovery
**What goes wrong:** flashing stock upstream firmware is phase-gated destructive verification; doing it without the recovery path written into the plan violates repo rules.
**How to avoid:** the hardware-run plan must include the recovery procedure verbatim (reflash Rust firmware via `just flash-monitor board=205 ...`, NVS reseed via existing seed flow) BEFORE the A/B task, plus required evidence fields (board, port, commits, board-info, logs, conclusion). Upstream build must not dirty `reference/esp-miner` (`just verify-reference` after).
