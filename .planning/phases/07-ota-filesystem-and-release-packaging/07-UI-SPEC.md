# Phase 07 - UI Design Contract

> Visual and interaction contract for OTA, filesystem, recovery, and release packaging surfaces. Generated by gsd-ui-researcher, verified by gsd-ui-checker.

***

## Design System

Phase 7 does not create a new frontend, replace AxeOS, rewrite Angular, or introduce a component registry. The design system for this phase is the existing upstream AxeOS static asset and recovery contract plus the update/status behavior those assets expect from `/api/system/OTA`, `/api/system/OTAWWW`, static file serving, and `/recovery`.

| Property | Value |
|----------|-------|
| Tool | none |
| Preset | not applicable |
| Component library | Existing upstream AxeOS Angular 18 asset tree with PrimeNG 17, PrimeFlex 3, and generated OpenAPI client models |
| Icon library | PrimeIcons (`pi ...` classes) for AxeOS; no icon library in the embedded recovery page |
| Font | AxeOS: `Nippo-Regular` from `/assets/fonts/Nippo-Regular.woff2`, weight 400. Recovery: browser monospace using `courier new`, weight 400 |

Source: `.planning/REQUIREMENTS.md` REL-01 through REL-08, `.planning/phases/07-ota-filesystem-and-release-packaging/07-CONTEXT.md` D-06 through D-17, `.planning/phases/07-ota-filesystem-and-release-packaging/07-RESEARCH.md`, `reference/esp-miner/main/http_server/axe-os/package.json`, and `reference/esp-miner/main/http_server/recovery_page.html`.

Hard constraints:

- Do not add shadcn, Tailwind, Radix, React, Next.js, Vite, Solid, or a new frontend framework in Phase 7.
- Do not redesign AxeOS screens, navigation, cards, colors, typography, spacing, icons, update modal, file upload controls, or dashboard layout.
- Do not replace the upstream update page contract. Preserve the existing `Update`, `Update AxeOS`, `Update Firmware`, release check, file upload, progress modal, and update result behaviors where static assets are served or fixture-tested.
- Do not replace the upstream recovery behavior with a marketing, onboarding, or docs page. `/recovery` is an operator recovery tool for uploading `www.bin` when AxeOS is unavailable.
- Do not claim `/api/system/OTAWWW` success or static-update parity unless implementation includes hardware/interruption/recovery evidence. If full OTAWWW parity is not proven, keep the visible behavior fail-closed and document REL-03 as a V1 gap.
- Preserve Phase 5's decision that V1 targets AxeOS asset/API compatibility, not a Rust-owned Angular replacement.

## Spacing Scale

Declared values (must be multiples of 4):

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px | Icon gaps, inline status spacing |
| sm | 8px | Compact file-upload/status spacing |
| md | 16px | Default control spacing, modal padding, doc callouts |
| lg | 24px | Update card spacing and recovery content grouping |
| xl | 32px | Layout gaps between update/release/recovery groups |
| 2xl | 48px | Major section breaks in operator docs |
| 3xl | 64px | Page-level spacing in new docs-only examples |

Exceptions:

- Existing AxeOS PrimeFlex and PrimeNG rem-based spacing remains inherited. The current layout scale is 14px and existing `1rem` card/dialog spacing must not be normalized to the 8-point scale.
- Existing AxeOS cards can remain nested where upstream already nests update cards inside the page card. Do not introduce additional nested cards.
- Existing recovery page uses bare HTML flow, `<pre>` ASCII art, `<p>`, file input, and buttons. If re-authored for provenance, keep it single-column, sparse, and recovery-focused; do not add decorative layout.
- File input and button hit targets for any re-authored fallback/recovery surface must be at least 44px tall.

## Typography

Use exactly the existing AxeOS typography where assets are served or fixture-tested. Do not introduce new display styles.

| Role | Size | Weight | Line Height |
|------|------|--------|-------------|
| Body | 14px | 400 | 1.5 |
| Label | 14px | 400 | 1.5 |
| Heading | 28px | 500 | 1.2 |
| Display | 35px | 500 | 1.2 |

Recovery exception:

- Existing `/recovery` uses monospace `courier new`, browser default body sizing, weight 400, and a readable line height as an inherited upstream exception. If re-authored for provenance, keep the four declared AxeOS sizes above and use monospace only for the recovery ASCII/status treatment.
- Status strings such as `Starting...`, `Working (42%)`, `Finished...`, `Rebooting...`, `Protocol Error`, and `Write Error` must fit in short status containers and firmware buffers; keep public status labels under 20 characters when they map to `firmware_update_status`.

Source: `reference/esp-miner/main/http_server/axe-os/src/styles.scss`, `reference/esp-miner/main/http_server/axe-os/src/app/layout/styles/layout/_variables.scss`, `reference/esp-miner/main/http_server/axe-os/src/app/layout/styles/layout/_typography.scss`, `reference/esp-miner/main/http_server/recovery_page.html`, and `reference/esp-miner/main/http_server/http_server.c`.

## Color

Preserve the existing dark AxeOS theme. Do not introduce a new palette.

| Role | Value | Usage |
|------|-------|-------|
| Dominant (60%) | `#070D17` | AxeOS background, surface ground, surface section |
| Secondary (30%) | `#000000`, `#192730`, `#2e3744` | Recovery background, AxeOS borders, subdued panels, secondary surfaces |
| Accent (10%) | `#F80421` | Primary buttons, active toggles, highlights, sliders, progress values, selected checkboxes |
| Destructive | `#f48fb1` | Existing PrimeNG danger states and destructive/error affordances only |

Accent reserved for: primary button background, button hover family, toggle selected state, checkbox selected state, slider range/handle, progress bar value, active highlight, and focus ring. Accent is not a generic color for every link, file name, status string, or operator warning.

Recovery color exception:

| Role | Value | Usage |
|------|-------|-------|
| Recovery background | `#000000` | Embedded recovery page background |
| Recovery text | `#00ff00` | Recovery page text and ASCII heading |

Status color rule: color may reinforce progress, success, warning, or error states, but the visible text must carry the state. Do not rely on red/green alone for update status.

## Phase 7 Compatibility Contract

### Visual Hierarchy

The primary AxeOS focal point remains the existing update cards and non-closable progress modal. The recovery page focal point remains the `www.bin` upload control, the 60 second wait warning, and the response/restart controls; do not add decorative panels or competing hero content.

### Static Asset And Recovery Contract

| Surface | Required Phase 7 behavior | Visible client impact |
|---------|---------------------------|------------------------|
| `GET /` | Serve `index.html` from mounted `www` SPIFFS when filesystem is available. Prefer gzipped asset variants when present. | Existing AxeOS loads without a route or MIME-type redesign. |
| Static file lookup | Normalize requests, reject path traversal, serve only mounted SPIFFS assets, set cache headers for non-directory static assets, and prefer `.gz` when available. | Browser asset loading remains compatible and efficient without exposing arbitrary files. |
| Missing static file | Redirect to `/` using upstream-compatible `302 Temporary Redirect` behavior unless a later plan records a safer compatible exception. | AxeOS client-side routing and missing route behavior keep working. |
| `/recovery` | Always register an explicit recovery route. Serve embedded recovery HTML even when SPIFFS is missing or AxeOS assets are broken. | Owner has a recovery path when `www.bin` is absent, corrupt, or interrupted. |
| Filesystem unavailable | Logs and any status surface must state that SPIFFS/static assets are unavailable. Do not silently return broken static HTML. | Operator sees recovery behavior instead of an opaque failed admin UI. |
| API coexistence | `/api/*`, `/api/ws`, `/api/ws/live`, `/api/system/OTA`, and `/api/system/OTAWWW` must not be captured by static wildcard routes. | Existing API clients and update page calls reach API handlers, not static files. |

### Update Interaction Contract

| Surface | Required Phase 7 behavior | Visible client impact |
|---------|---------------------------|------------------------|
| AxeOS firmware upload | Existing update page accepts only a file named `esp-miner.bin`, opens a modal headed `Updating Firmware`, shows upload percentage when available, then shows success or error copy. | User sees the same firmware update flow expected from upstream AxeOS. |
| `/api/system/OTA` | Preserve private-network/origin gate, AP/APSTA rejection, streamed binary upload, progress/status labels, validation/activation error copy, success text response, reboot scheduling, and rollback/boot-validation evidence. | Upload success means the device is moving toward reboot/validation, not merely that bytes were received. |
| AxeOS static upload | Existing update page accepts only a file named `www.bin`, opens a modal headed `Updating AxeOS`, shows upload percentage when available, then reloads the page after success. | User sees static UI update behavior without a redesigned upload flow. |
| `/api/system/OTAWWW` implemented path | If implemented, preserve whole-`www` partition update behavior: AP/APSTA rejection, `www` partition lookup, too-large rejection, chunked erase/write, progress/status labels, success text response, recovery evidence, and interrupted-update evidence. | Static update can move beyond fail-closed only with evidence for recovery after interruption. |
| `/api/system/OTAWWW` gap path | If full OTAWWW parity is not proven, keep the route fail-closed and record REL-03 with owner, evidence, release impact, and follow-up. Use public failure text compatible with existing clients. | User cannot falsely believe the web UI partition was safely updated. |
| Recovery upload | Recovery page posts `www.bin` to `/api/system/OTAWWW`, shows `uploading...`, then displays response text and either `Upload successful!` or `Error uploading!`. | Recovery remains usable when the normal AxeOS app cannot load. |
| Recovery restart | Recovery page exposes an explicit `Restart` button that posts to `/api/system/restart` and displays the response. | Restart is operator-initiated after response, not hidden during upload. |

### Release And Operator Docs Contract

Operator-facing docs are part of the UI contract because they guide high-risk update/recovery actions.

| Docs Surface | Required Copy/Structure |
|--------------|-------------------------|
| Artifact list | Name each direct artifact: firmware app image, `www.bin`, factory/recovery image, update-only image if present, manifest v2, license inventory, provenance manifest, and install notes. |
| Normal local flow | Show `just build`, `just package`, `just flash board=205`, `just monitor`, and `just flash-monitor board=205` with expected output references. |
| Firmware OTA flow | Explain upload target `/api/system/OTA`, expected file name `esp-miner.bin`, reboot behavior, rollback/boot-validation evidence, and recovery steps if boot validation fails. |
| AxeOS/static flow | Explain `/api/system/OTAWWW` only if implemented with evidence. If gapped, state that clearly and route users to factory image/recovery docs. |
| Recovery flow | Explain `/recovery`, `www.bin`, the 60 second wait, not restarting before response, and when to use `Restart`. |
| Large erase/interruption | Explain that static partition writes can leave AxeOS unavailable and that `/recovery` is the required fallback. |
| Release caveats | Separate package evidence, live OTA evidence, live recovery evidence, interrupted-update evidence, license review, and provenance review. |

Tone for operator docs:

- Use direct imperative steps.
- Include exact command names and route names.
- Do not bury safety warnings in long paragraphs.
- Do not claim "verified", "safe", "release-ready", or "parity" unless the matching evidence row exists.

### Evidence Visibility Contract

| Evidence State | UI/Docs Claim Allowed |
|----------------|-----------------------|
| Package-only | May say artifacts were generated and checksums were recorded. Must not say OTA or recovery was verified. |
| Host/unit tested | May say planner/route/static decisions match expected cases. Must not say hardware update was verified. |
| Firmware smoke | May say route served or upload path executed on firmware under the captured conditions. Must name board, port, commit, and artifact. |
| Hardware/interrupted-update evidence | May support `verified` OTAWWW or rollback/recovery claims only when evidence records the interruption/rollback/recovery result. |
| Explicit REL-03 gap | Must say static update remains unavailable or unverified for V1 release candidate scope and name the owner/follow-up path. |
