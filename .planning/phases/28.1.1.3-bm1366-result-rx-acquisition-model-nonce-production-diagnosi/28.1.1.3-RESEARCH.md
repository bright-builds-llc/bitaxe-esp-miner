## Common Pitfalls

### Pitfall 1: Trusting `partial_frame=0` From Flooded Logs
**What goes wrong:** Planner skips clear_rx hypothesis or picks wrong lever.
**Why it happens:** Per-chunk `asic_uart_trace=rx` + hex under phase27 bridge overwhelms monitor capture; `rx_idle` also missing despite enabled uart_trace.
**How to avoid:** Emit first-class comparator fields; add compact summary markers; treat raw uart_trace counts as instrumentation-limited.
**Warning signs:** Thousands of `result_read_attempt` with near-zero `rx_idle`/`partial_frame` while job `asic_uart_trace=tx` still appears.

### Pitfall 2: Longer Poll Slice Without Raising Socket Clamp
**What goes wrong:** A/B appears to change `MAX_POLL_SLICE_MS` but runtime still polls 100 ms.
**Why it happens:** `poll_timeout_ms = slice_ms.min(SOCKET_TIMEOUT_MS)` and `SOCKET_TIMEOUT_MS=100`.
**How to avoid:** Prefer long-block investigation path that bypasses the clamp; if testing longer poll, change clamp deliberately and document both.
**Warning signs:** A/B logs still show `poll_timeout_ms=100`.

### Pitfall 3: Misreading `result_read_attempt` As TX
**What goes wrong:** “Remove result probes” as the A/B.
**Why it happens:** Name looks like a TX probe; it is RX poll logging. [VERIFIED: production.rs; 28.1.1.2 Pitfall 1]
**How to avoid:** Keep D-05; A/B must change RX timeout/accumulation behavior, not suppress the log line.

### Pitfall 4: Partial-Frame `clear_rx` Discard (carry-forward)
**What goes wrong:** Short poll gets a fragment, `clear_rx` flushes the ring, nonce frame lost; maps to `Pending`.
**Why it happens:** `uart.rs` clears on partial; `production.rs` maps `Err` to `Pending`. [VERIFIED: uart.rs:158-176, production.rs:238-240]
**How to avoid:** Long-block accumulate within one `try_read_exact(11, 10000)` matches upstream and also reduces fragment risk; retain-partial is a secondary lever only if long-block fails and compact counters show high `partial_frame`.
**Warning signs:** Compact `partial_frame` spikes with zero `result_correlated`.

### Pitfall 5: Recommender Still Preferring Falsified Register-Read Poll
**What goes wrong:** Baseline report recommends `match_upstream_register_read_poll` again because chip_id asymmetry remains.
**Why it happens:** compare.mjs recommender prioritizes chip_id flood before `result_rx_acquisition_model`. [VERIFIED: compare.mjs:265-282]
**How to avoid:** For 28.1.1.3, force A/B label in diagnostic wrapper (as 28.1.1.2 did); update recommender so after falsified register-read, jobs+result_read+!correlate → `result_rx_acquisition_model` without requiring `partial_frame >= 5`.
**Warning signs:** Diagnostic auto-selects register-read poll again.

### Pitfall 6: Overclaiming Phase 30 / Job Bytes
**What goes wrong:** Checklist promotion or job-frame patches creep in.
**Why it happens:** STR-09/CFG-07/ASIC-11 IDs appear on the phase.
**How to avoid:** Blocker-closure input only; `phase30_promotion_input: pending`; no job-byte edits.
**Warning signs:** PRs touching `docs/parity/checklist.md` verified rows or job builders.

### Pitfall 7: Starving Stratum Pump With Naive 10s Block
**What goes wrong:** Long-block A/B blocks the shared FreeRTOS task for up to 10s, delaying socket reads / job dispatch.
**Why it happens:** Rust couples result poll and Stratum pump; upstream uses a dedicated result task.
**How to avoid:** Accept as investigation-only tradeoff; log `poll_timeout_ms=10000`; if A/B improves correlate but regresses pool liveness, note for a follow-up dedicated-task design (out of scope unless needed). Do not add a second flag for task split in this phase.
**Warning signs:** Fake-pool disconnects / missing notifies during long-block run.
