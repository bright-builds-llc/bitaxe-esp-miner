# Phase 05 - UI Design Contract

> Visual and interaction contract for existing AxeOS/static asset compatibility. Generated by gsd-ui-researcher, verified by gsd-ui-checker.

---

## Design System

Phase 5 does not create a new frontend, rewrite Angular, or introduce a component registry. The design system for this phase is the existing upstream AxeOS asset contract plus the API/log/WebSocket behavior those assets expect.

| Property | Value |
|----------|-------|
| Tool | none |
| Preset | not applicable |
| Component library | Existing upstream AxeOS Angular 18 asset tree with PrimeNG 17, PrimeFlex 3, and generated OpenAPI client models |
| Icon library | PrimeIcons (`pi ...` classes) |
| Font | `Nippo-Regular` from `/assets/fonts/Nippo-Regular.woff2`, weight 400 |

Source: `.planning/REQUIREMENTS.md` API-09, `.planning/phases/05-axeos-api-logs-and-telemetry/05-CONTEXT.md` D-25 through D-26, and `reference/esp-miner/main/http_server/axe-os/package.json`.

Hard constraints:

- Do not add shadcn, Tailwind, Radix, React, Next.js, Vite, Solid, or a new frontend framework in Phase 5.
- Do not redesign AxeOS screens, navigation, cards, colors, typography, spacing, icons, or dashboard layout.
- Do not implement Phase 7 OTA, OTAWWW, SPIFFS image production, release packaging, or static-asset update flows in Phase 5.
- Do not make voltage, fan, thermal, power, or unsafe mining UI states appear verified or active without Phase 6 hardware evidence.

## Phase 5 Compatibility Contract

### Route And Interaction Contract

All route behavior is part of the UI contract because existing AxeOS clients render state directly from these responses.

| Surface | Required Phase 5 behavior | Visible client impact |
|---------|---------------------------|------------------------|
| `GET /api/system/info` | Return upstream-compatible full-state JSON with exact field names, casing, units, defaults, and mixed encodings such as `ASICModel`, `hashRate_1m`, `fanspeed`, `fanrpm`, numeric bool-like fields, and JSON booleans where upstream uses booleans. Use safe blocked/unavailable values for Phase 6 hardware-control fields. | Dashboard, topbar pause state, pool/network/settings pages, swarm discovery, and `/api/ws/live` baseline do not break on missing or renamed fields. |
| `GET /api/system/asic` | Return upstream-compatible ASIC settings shape: `ASICModel`, `deviceModel`, `swarmColor` when known, `asicCount`, `defaultFrequency`, `frequencyOptions`, `defaultVoltage`, and `voltageOptions`. Ultra 205/BM1366 facts come from typed Rust catalog/status data. | Settings and swarm views can display board/ASIC controls without treating deferred hardware effects as active. |
| `GET /api/system/statistics?columns=...` | Accept optional column selection, always include `timestamp`, return `currentTimestamp`, `labels`, and `statistics` arrays. When history is unavailable, return an empty or minimal compatible series with evidence; never fake historical mining data. | Charts can render or show no points without schema failure. |
| `GET /api/system/scoreboard` | Return an array of entries containing only server-owned fields: `difficulty`, `job_id`, `extranonce2`, `ntime`, `nonce`, `version_bits`. Do not add authoritative server-side `rank` or `since`; existing AxeOS derives those. | Scoreboard table sorts and displays entries every 5000 ms without contract drift. |
| `PATCH /api/system` | Parse JSON, validate all known REST-named settings atomically, ignore unknown fields, reject the whole patch when any known field is invalid, write nothing on rejection, persist/reload before success when the adapter can prove it, and return an empty success body. | Existing edit/settings forms see upstream-compatible save/reject behavior and generic errors. |
| `GET /api/system/logs` | Return `text/plain` with `Content-Disposition: attachment; filename="bitaxe-logs.txt"`, chunk retained log data from the absolute beginning, and end with an empty chunk. | Logs page downloads `bitaxe-logs.txt` and shows success/failure toasts without special casing. |
| `GET /api/ws` | Upgrade to WebSocket for raw log text chunks. New clients start at the current log-buffer end, not at retained history. Send chunks only when clients are connected and new log data exists. | Logs page streams live logs, filters complete lines, and keeps the latest 256 rendered lines. |
| `GET /api/ws/live` | Upgrade to WebSocket for telemetry. On connect send `{"event":"update","data":...}` with full current state. Then send changed-field diffs every 500 ms only when values change and clients are connected. Reset baseline while no clients are connected. | LiveDataService merges updates without polling churn and falls back to polling when disconnected. |
| `POST /api/system/pause` | Set typed mining activity state to paused, return `{"message":"Mining paused"}`, and update API-visible `miningPaused` without making work submission ready. | Topbar pause toggle and dashboard paused banner stay coherent. |
| `POST /api/system/resume` | Clear typed paused state, return `{"message":"Mining resumed"}`, and preserve Phase 4 mining-loop gates. | Resume appears successful without bypassing safety gates. |
| `POST /api/system/restart` | Return `{"message":"System will restart shortly."}` before scheduling the restart effect. Tests must prove response-before-effect ordering. | Existing restart callers can show success before the socket/device drops. |
| `POST /api/system/identify` | Toggle identify mode. When enabling, expose 30 seconds of visible identify duration and return `{"message":"The device says \"Hi!\" for 30 seconds."}`. When disabling, return `{"message":"The device no longer says \"Hi!\"."}`. | System page identify action has the same visible feedback as upstream. |
| `POST /api/system/blockFound/dismiss` | Clear `showNewBlock`, preserve `blockFound`, and return `blockFound`, `showNewBlock`, and `message`. | Block-found notification dismissal remains client-compatible. |
| `/api/system/OTA` and `/api/system/OTAWWW` | Phase 5 must not accept or apply uploads. If existing assets invoke these routes, return safe unsupported/deferred failure rather than success, and record that Phase 7 owns implementation. | Update page may render an error, but cannot falsely report an applied firmware or AxeOS update. |
| Static files and `/recovery` | Preserve route expectations and fixture-test existing AxeOS asset calls. If filesystem/static packaging is unavailable in Phase 5, recovery/static fallback must remain safe and must not claim OTA support. | Existing assets remain usable for V1 API/log/WebSocket administration; release packaging stays Phase 7. |

### Log And Telemetry Contract

| Requirement | Contract |
|-------------|----------|
| Retained buffer | Model the upstream retained log ring buffer at 512 KiB. Clamp old readers to retained contents and resync to the next newline within a bounded 4 KiB scan when clamped. |
| Log chunking | Use 4096 byte chunks for retained download and live WebSocket draining where practical. Preserve newline-aware reads for complete-line UI rendering. |
| Client cap | Preserve the upstream client cap of 10 WebSocket clients unless implementation planning records a stricter safe limit with compatibility evidence. |
| No-client behavior | `/api/ws` keeps its cursor at the current end while no log clients are connected; `/api/ws/live` deletes/clears its comparison baseline while no live clients are connected. |
| Reconnect behavior | Existing clients may reconnect every 5000 ms and fall back to `GET /api/system/info` polling every 5000 ms when visible or 60000 ms when hidden. API behavior must not make that fallback noisy or state-inconsistent. |

### Response-Visible State Contract

| State | Required response-visible behavior |
|-------|------------------------------------|
| Safe blocked mining | Surface Phase 4 blocked reasons such as `hardware_evidence_ack_missing` through compatible fields/statuses. Do not convert blocked mining into active mining. |
| Paused mining | `miningPaused` must be present and boolean in `/api/system/info` and live telemetry. Pause/resume command responses must match upstream copy. |
| Unavailable safety telemetry | Voltage, fan, thermal, power, current, and related Phase 6 fields must use upstream-compatible safe defaults or unavailable/deferred values. Do not invent live readings. |
| Empty statistics | Empty or minimal `statistics` data is allowed only with documented evidence. The shape must remain compatible. |
| Empty scoreboard | Return an empty array `[]`, not `null` and not an object. |
| Unknown API route | Return HTTP 404 with JSON body `{"error":"unknown route"}` for unhandled `/api/*` routes when static filesystem mode is active. |
| Unauthorized network/origin | Return HTTP 401 with public text `Unauthorized` for requests rejected by the private-network/AP-origin gate. |
| Invalid settings JSON | Return HTTP 400 with public text `Invalid JSON`. |
| Invalid known settings | Return HTTP 400 with public text `Wrong API input`; write nothing. |
| Oversized settings body | Fail visibly and do not write settings. Preserve upstream-compatible generic server failure behavior unless planning proves a safer compatible error. |

### Static Asset Compatibility Evidence

Phase 5 evidence must prove compatibility against existing AxeOS clients without a redesign.

Required fixture/smoke coverage:

- Existing `SystemApiService` route calls remain satisfied for V1 administration: info, ASIC settings, statistics, scoreboard, logs, settings PATCH, pause, resume, restart, identify, block-found dismiss.
- Existing `LiveDataService` receives initial full live telemetry and subsequent diff updates through `/api/ws/live`.
- Existing `WebsocketService` receives raw log text through `/api/ws`.
- Existing Logs page behavior remains compatible: heading, case-sensitive filter, pause/resume scrolling, clear local logs, download `bitaxe-logs.txt`, and latest 256 rendered lines.
- Existing Scoreboard page behavior remains compatible: 5000 ms refresh, client-derived rank/since, sortable columns, and empty-array handling.
- Existing Update page routes are explicitly deferred: OTA and OTAWWW requests cannot succeed in Phase 5 and are documented as Phase 7-owned.
- Existing static/recovery route behavior is fixture-tested or explicitly marked as Phase 7 packaging evidence pending. Do not introduce Angular build pipeline work unless planning proves it is bounded and necessary.

### API Comparison Evidence Contract

| Evidence type | Required proof |
|---------------|----------------|
| Schema compatibility | Route coverage and required property checks against `reference/esp-miner/main/http_server/openapi.yaml` for V1 Phase 5 routes. |
| Captured response compatibility | Representative success/error JSON fixtures for info, ASIC, statistics, scoreboard, settings PATCH, logs, WebSocket envelopes, and commands. |
| Firmware smoke | Real or emulator-backed HTTP/WebSocket smoke for representative success and error cases. Mark hardware-control and safety telemetry below verified unless Phase 6 hardware evidence exists. |
| Static asset compatibility | Fixture or smoke proving existing AxeOS service calls can administer V1 API/log/WebSocket surfaces without an Angular rewrite. |
