## Summary

Phase 28.1.1.1 already proved aligned source-work BM1366 job fields match 11/11 and below-job semantic events match 12/12, while Rust still lacks `result_correlated` and fake-pool submit. The leftover count asymmetry (`upstream chip_id_read_tx≈2015` vs `Rust≈3`, `upstream result_read_attempt=0` vs `Rust≈1458`) is real evidence, but it is easy to misread. Verified code shows Rust `result_read_attempt` is an RX-poll log line from `try_read_production_result`, not a TX-driven result-register probe. Upstream steady-state result acquisition is passive `receive_work` → `SERIAL_rx` in `ASIC_result_task`. Upstream’s high `chip_id_read_tx` count is the hashrate-monitor `BM1366_read_registers` loop (CMD_READ header `0x52`, ~6 regs/s), which the existing comparator collapses into `chip_id_read_tx`.

## Common Pitfalls

### Pitfall 1: Misreading `result_read_attempt=0` on upstream
**What goes wrong:** Planner assumes upstream never polls results.
**Why it happens:** Upstream has no equivalent log line; it blocks in `SERIAL_rx` for up to 10 s.
**How to avoid:** Comparator should label upstream `result_read_attempt=0` as `expected_no_marker`, not `missing_rx_poll`.
**Warning signs:** Recommendations that try to “add result_read_attempt to upstream” or “remove Rust RX polling entirely” without measuring nonce RX.

### Pitfall 2: Collapsing all `0x52` CMD_READ into `chip_id_read_tx`
**What goes wrong:** Treats hashrate-monitor register polling as chip-detect only.
**Why it happens:** Classifier uses TX header `0x52` without register address discrimination. [VERIFIED: below-job comparator `classifyFrame` header===0x52; upstream `TYPE_CMD|GROUP_ALL|CMD_READ=0x52`]
**How to avoid:** Result-path comparator should still count `chip_id_read_tx` (D-05) but recommend label `match_upstream_register_read_poll` when upstream≫Rust; optionally sub-count by register byte when present in logs.
**Warning signs:** “Rust is missing chip detect” after init already succeeded.

### Pitfall 3: Under-counting upstream `rx_result_frame`
**What goes wrong:** Upstream `rx_result_frame=0` looks like no RX while shares submit.
**Why it happens:** Upstream RX hex only prints when `BM1366_SERIALRX_DEBUG` (etc.) is enabled; submit path does not require those logs. [VERIFIED: serial.c:81-88]
**How to avoid:** Prefer fake-pool `submit_observed` + Rust `result_correlated` as success gates; treat upstream RX count as instrumentation-dependent.
**Warning signs:** Forcing upstream RX parity before enabling RX debug in scratch captures.

### Pitfall 4: Partial-frame `clear_rx` dropping nonces
**What goes wrong:** Short 100 ms polls + partial read clear the UART buffer and discard in-flight nonce bytes.
**Why it happens:** `uart.rs` clears RX on partial; `production.rs` maps that Err to `Pending`. [VERIFIED: uart.rs:158-176, production.rs:192-194]
**How to avoid:** After register-poll A/B, if still gapped, investigate partial_frame / clear_rx as second hypothesis (D-07 item 1). Do not lead with this as default patch without comparator evidence.
**Warning signs:** `asic_uart_trace=partial_frame` spikes with zero `result_correlated`.

### Pitfall 5: Recommending falsified investigation modes
**What goes wrong:** Old below-job `recommendInvestigation` returns `post_max_baud_delay_2000` whenever jobs+result_read exist without correlation.
**Why it happens:** Heuristic predates the falsifying A/B. [VERIFIED: compare.mjs:314-319 + final evidence]
**How to avoid:** Result-path recommender must not default to `post_max_baud_delay_2000`. Prefer `match_upstream_register_read_poll`, `result_rx_acquisition_model`, or `none`.
**Warning signs:** A/B scripts auto-selecting post_max_baud again.

### Pitfall 6: Overclaiming Phase 30 promotion
**What goes wrong:** Checklist rows flip to `verified` from this phase.
**Why it happens:** STR-09/CFG-07/ASIC-11 map to Phase 30 for promotion.
**How to avoid:** Emit `phase30_promotion_input: pending|ready` category only; never edit checklist verified status here.
**Warning signs:** PRs touching `docs/parity/checklist.md` verified rows in 28.1.1.2.
